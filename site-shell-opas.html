Sivuston yhteinen site shell opas

Tämä sivu toimii projektin kanonisena opasteena yhteiselle headerille, footerille, fontille, väriteemalle ja päivitysprosessille. Tätä sivua linkitetään lähteisiin, jotta kaikki tulevat HTML-sivut pysyvät samassa visuaalisessa rungossa.

Siirry toimintamalliin Siirry markerimalliin

Tämä ei ole varsinainen sisältösivu. Tämä on runko, ohje ja yhteinen visuaalinen lähde.

Tämän sivun tarkoitus

  • määrittää yhteinen yläpalkki
  • määrittää yhteinen navigaatio
  • määrittää yhteinen footer
  • määrittää yhteinen väriteema ja typografia
  • määrittää miten vanhoja sivuja päivitetään
  • määrittää miten uusia poikkeamia estetään

Yksi sivu toimii lähteenä. Muut sivut seuraavat sitä. Näin GitHub Pages sivustosta syntyy yhtenäinen lehtiö, ei irrallisten HTML-sivujen kasa.

1. Kanoninen site shell

  • langbar on ylin rivi
  • site-nav on sivujen välinen navigaatio
  • hero header on visuaalinen avaus
  • footer-cta on sivun loppunosto
  • footer on viimeinen yhteinen alatunniste
Sisältö saa vaihtua sivukohtaisesti. Site shell ei vaihdu sattumalta, vaan hallitusti.

2. Mitä ei rikota

  • sama fonttiperhe
  • sama väritoken-rakenne
  • sama padding- ja korttirytmi
  • sama headerin ja footerin rakenne
  • sama sivuleveys ja sama korttityyli
Yksittäistä sivua ei “parannella” eri suuntaan ilman että koko shell päivitetään hallitusti.

3. Päivitysmalli vanhoille sivuille

1. VERIFY

Varmista repo, branch, nykyiset HTML-sivut ja käytössä oleva header-rakenne.

2. DIFF

Vertaa vanhaa sivua tähän opassivuun. Etsi vain site shelliin liittyvät erot.

3. PATCH

Päivitä vain langbar, site-nav, hero-shell ja footer-rakenne. Älä sekoita sisältöön.

4. VERIFY

Avaa sivu selaimessa, tarkista navigaatio, aktiivinen linkki, mobiilirakenne ja footer.

4. Markerimalli yhteisille lohkoille

Jokaisessa päivitettävässä sivussa yhteiset osat kannattaa rajata samoilla markkereilla. Silloin terminalista tai skriptillä voidaan päivittää vain yhteiset lohkot ilman että sisältö hajoaa. 

<!-- SHARED:LANGBAR:START -->
... langbar tähän ...
<!-- SHARED:LANGBAR:END -->

<!-- SHARED:SITE_NAV:START -->
... site-nav tähän ...
<!-- SHARED:SITE_NAV:END -->

<!-- SHARED:FOOTER_CTA:START -->
... footer-cta tähän ...
<!-- SHARED:FOOTER_CTA:END -->

<!-- SHARED:FOOTER:START -->
... footer tähän ...
<!-- SHARED:FOOTER:END -->
Markerimalli on käytännöllinen, koska se rajaa päivityksen tarkasti vain yhteisiin lohkoihin.

5. Milloin luodaan uutta

Uutta yhteistä päivitysmekanismia ei luoda automaattisesti. Ensin tarkistetaan, löytyykö reposta jo valmiiksi yhteinen CSS, yhteinen JS, layout-pohja tai aiempi header-komponentti.

6. Milloin vain päivitetään vanhaa

Jos reposta löytyy jo olemassa oleva yhteinen navigaatiomalli, yhteinen layout-tiedosto, tai edes toistuva HTML-rakenne, sitä käytetään lähtökohtana. Tämä opassivu toimii silloin vertailupohjana, ei rinnakkaisena toisena totuutena.

7. Terminal-first audit checklist

  • mikä repo on oikea
  • mikä branch on aktiivinen
  • mitkä HTML-sivut kuuluvat tähän GitHub Pages kokonaisuuteen
  • onko navissa jo uusin sivu
  • onko header kahtena rivinä kuten pitää
  • onko footer kaikissa sama

8. Site shellin ydinajatus

Tämä sivu on visuaalinen standardi, rakenteellinen opas ja muutosten kontrollipiste. Kun sivuja tulee lisää, ensin päivitetään tämä standardi tai varmennetaan että se on edelleen oikea. Sen jälkeen muut sivut synkronoidaan hallitusti samaan runkoon.

Yksi yhteinen runko tekee kokonaisuudesta uskottavan

Kun header, navigaatio, typografia ja footer elävät saman mallin mukaan, GitHub Pages sivustosta tulee eheä julkaisu eikä irrallisten demojen ketju.
